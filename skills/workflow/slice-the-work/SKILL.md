# Slice The Work

Produce the fewest small slices that cover the accepted PRD.

## Vertical rule

A normal vertical slice delivers a usable product capability through every layer that capability requires. Express it as `<actor> can <do or observe something valuable>`. It need not always touch frontend, backend, and database: an API product may have no UI, and a client-only capability may need no database. But never split one capability into separate schema, API, UI, and test tickets.

This exposes integration errors early and produces useful evidence. Split by outcome, journey, rule, or independently verifiable risk, not architecture or team.

Product-feature examples:

- **Appointment booking:** A salon publishes available slots and customers book without double-booking; scheduling rules, persistence, API, UI, confirmations, and concurrency tests ship together.
- **Notification preferences:** A user turns off a notification category and it stays off across devices; preferences, API, UI, delivery enforcement, account isolation, and tests ship together.
- **Offline playlist:** A listener downloads a playlist and plays it without a connection; download state, local storage, sync, UI, retry behavior, and tests ship together.

Anti-example: separate notification-settings table, endpoint, toggle, and test issues.

## Named exceptions

When vertical delivery is unsafe or impossible, record `Exception: <name>`, its rationale, and its consumer or exit criterion.

- **Enabler/refactor:** shared contract or infrastructure required by multiple imminent slices that cannot safely be introduced in the first consumer.
- **Discovery spike:** time-boxed uncertainty ending in evidence or a decision, not production.
- **Safety/sequence:** migration, rollback, observability, or rollout that explicit risk or ordering constraints require isolating.
- **External/HITL gate:** owned access, approval, vendor, or human action.

Use the smallest unblocker and link the slices it enables. "Backend first" and team convenience are not exceptions.

## Process

1. Extract requirements, outcomes, decisions, constraints, non-goals, and evidence from the PRD.
2. Draft dependency-ordered slices and candidate execution waves, preserving requirement and decision IDs.
3. Test each: Is its outcome observable, are all required layers included, and can it be verified independently? If not, redraw it or name an exception.
4. Label `AFK` or `HITL`, check coverage, and obtain granularity approval unless told to proceed.
5. After approval, invoke `$model-route Recommend` once for the set. Write each route, container, and wave into its issue, then publish.

```md
# <slice>
Type | parent PRD | blockers | wave:
Outcome | requirements/decisions:
Change | evidence:
Exception: none | <name, rationale, consumer/exit>
Out of scope:
Execution route / container: <generated by $model-route>
```

Finish when every normal issue passes the vertical test, every exception is bounded, all requirements and decisions are covered, dependencies and waves are explicit, and every issue is routed.

Return ordered refs and routing status. When any route uses child tasks, return `$orchestrate-the-work`; otherwise return `$deliver-the-slice`.
